# Long-Running Operations (LRO) {#lro-operations}

Длительная операция (**Long-running operation**, cокращение: **LRO**) — это выполнение команды, результат которой невозможно или нецелесообразно вернуть в рамках обычного синхронного запроса.

К длительным относятся операции, которые:

- выполняются дольше допустимого времени синхронного HTTP, ConnectRPC или gRPC-запроса;
- требуют взаимодействия с несколькими сервисами;
- включают ожидание внешней системы;
- состоят из нескольких этапов;
- должны переживать перезапуск сервиса;
- требуют повторных попыток;
- могут быть отменены пользователем;
- должны предоставлять прогресс выполнения;
- завершаются после асинхронного подтверждения;
- изменяют несколько ресурсов или внешних систем;
- требуют компенсирующих действий при ошибке.

{% cut "Примеры длительных операций" %}

- создание организации, проекта или сервиса с подготовкой инфраструктуры;
- импорт большого объёма данных;
- экспорт отчёта;
- массовое изменение ресурсов;
- перенос данных;
- удаление организации со связанными ресурсами;
- развёртывание сервиса;
- выпуск сертификата;
- синхронизация с внешним провайдером;
- обучение и публикация модели;
- выполнение многоэтапного процесса аутентификации;
- изменение конфигурации, требующее согласования и применения на нескольких компонентах.

{% endcut %}

Длительная операция не является отдельным бизнес-ресурсом. Она представляет состояние выполнения команды над бизнес-ресурсом.

## Цели

Механизм длительных операций должен обеспечивать:

1. устойчивое выполнение команд независимо от времени жизни клиентского соединения;
2. единообразный интерфейс для всех модулей платформы;
3. получение текущего состояния и результата операции;
4. отображение прогресса;
5. безопасное повторение клиентских запросов;
6. повторные попытки при временных ошибках;
7. отмену операции, когда это допустимо;
8. восстановление после перезапуска компонентов;
9. аудит действий и переходов состояния;
10. трассировку выполнения между сервисами;
11. ограничение параллелизма и нагрузки;
12. поддержку частично завершённых процессов;
13. выполнение компенсирующих действий;
14. сохранение истории завершённых операций.

## Когда использовать длительную операцию

Длительная операция должна использоваться, если выполняется хотя бы одно из условий:

- ожидаемое время выполнения превышает 3-5 секунд;
- требуется более одного удалённого вызова;
- используется внешняя система с непредсказуемым временем ответа;
- операция включает асинхронное событие или обратный вызов;
- операция должна продолжаться после отключения клиента;
- операция требует повторных попыток;
- операция включает изменение нескольких агрегатов;
- операция может быть поставлена в очередь;
- необходим пользовательский прогресс;
- необходимо предоставить пользователю возможность отмены;
- выполнение зависит от ручного подтверждения;
- результат формируется отдельным вычислительным заданием.

Длительная операция не должна использоваться для простого сокрытия медленного API. Если команда может быть выполнена атомарно и быстро, она должна оставаться синхронной.

## Основной подход

Платформа использует модель асинхронного запуска:

1. клиент отправляет команду;
2. сервис проверяет запрос;
3. сервис создаёт запись операции;
4. сервис фиксирует начало бизнес-процесса;
5. клиент получает объект `Operation`;
6. выполнение продолжается независимо от соединения клиента;
7. клиент периодически запрашивает состояние операции либо получает обновления через поток событий;
8. после завершения операция содержит результат или ошибку.

Для публичного API используется контракт, совместимый с `google.longrunning.Operation`.

Для надёжного выполнения многоэтапных процессов используется Temporal.

Для публикации событий используется Transactional Outbox.

## Модель операции

Каждая операция имеет глобально уникальное имя.

Пример:

```text
operations/01JZ7A6K4WQH8Y3X2D5N9M1P7C
```

Рекомендуемый формат имени:

```text
operations/{operation_id}
```

Внутренние операции отдельных модулей могут дополнительно содержать пространство имён:

```text
resource-manager/operations/{operation_id}
provisioning/operations/{operation_id}
identity/operations/{operation_id}
```

Публичный API должен использовать единообразный формат, если разделение пространств имён не требуется для маршрутизации.

### Структура операции

```proto
message Operation {
  string name = 1;
  google.protobuf.Any metadata = 2;
  bool done = 3;

  oneof result {
    google.rpc.Status error = 4;
    google.protobuf.Any response = 5;
  }
}
```

Где:

- `name` — уникальное имя операции;
- `metadata` — текущее состояние и прогресс;
- `done` — признак завершения;
- `response` — успешный результат;
- `error` — итоговая ошибка.

До завершения операции поля `response` и `error` отсутствуют.

## Метаданные операции

Все операции платформы должны использовать общую базовую структуру метаданных.

```proto
message OperationMetadata {
  OperationState state = 1;
  string stage = 2;
  string message = 3;
  uint32 progress_percent = 4;

  google.protobuf.Timestamp create_time = 5;
  google.protobuf.Timestamp start_time = 6;
  google.protobuf.Timestamp update_time = 7;
  google.protobuf.Timestamp end_time = 8;

  string resource_name = 9;
  string requested_by = 10;
  bool cancel_requested = 11;
  bool cancellable = 12;

  uint32 attempt = 13;
  uint32 completed_steps = 14;
  uint32 total_steps = 15;

  repeated OperationWarning warnings = 16;
}
```

### Обязательные поля

Каждая операция должна предоставлять:

- текущее состояние;
- текущий этап;
- пользовательское сообщение;
- время создания;
- время последнего обновления;
- признак возможности отмены;
- ссылку на создаваемый или изменяемый ресурс, если она известна.

### Прогресс

`progress_percent` принимает значения от `0` до `100`.

Прогресс должен отражать фактическое состояние, а не время, прошедшее с момента запуска.

Если точный процент определить невозможно, допускается:

- не заполнять `progress_percent`;
- использовать количество выполненных этапов;
- отображать неопределённый прогресс на стороне клиента.

Прогресс не должен уменьшаться, кроме случаев, когда операция была перезапущена с новым планом выполнения. Такой случай должен фиксироваться в аудите.

### Этап

`stage` содержит стабильный машинно-читаемый код:

```text
VALIDATING
PLANNING
ALLOCATING
PROVISIONING
CONFIGURING
WAITING_EXTERNAL_SYSTEM
VERIFYING
FINALIZING
ROLLING_BACK
COMPLETED
```

`message` содержит человекочитаемое описание:

```text
Ожидание подтверждения создания кластера провайдером
```

Клиент не должен строить бизнес-логику на основе поля `message`.

## Состояния операции

```proto
enum OperationState {
  OPERATION_STATE_UNSPECIFIED = 0;
  PENDING = 1;
  RUNNING = 2;
  WAITING = 3;
  CANCELLING = 4;
  SUCCEEDED = 5;
  FAILED = 6;
  CANCELLED = 7;
}
```

### PENDING

Операция создана, но фактическое выполнение ещё не началось.

Причины:

- ожидание свободного исполнителя;
- ограничение параллелизма;
- ожидание квоты;
- отложенный запуск;
- ожидание блокировки ресурса.

### RUNNING

Выполняется активный этап операции.

### WAITING

Операция ожидает внешнее событие:

- ответ провайдера;
- подтверждение пользователя;
- поступление события;
- окончание заданного периода;
- снятие блокировки;
- ручное согласование.

`WAITING` не означает ошибку.

### CANCELLING

Получен запрос на отмену. Исполнитель завершает текущий безопасный этап и выполняет необходимые компенсирующие действия.

### SUCCEEDED

Операция успешно завершена. Поле `response` заполнено.

### FAILED

Операция завершена с ошибкой. Поле `error` заполнено.

### CANCELLED

Операция отменена. Частично созданные изменения должны быть либо компенсированы, либо явно отражены в результате и состоянии целевого ресурса.

## Допустимые переходы состояний

```text
PENDING → RUNNING
PENDING → CANCELLING
PENDING → CANCELLED
PENDING → FAILED

RUNNING → WAITING
RUNNING → CANCELLING
RUNNING → SUCCEEDED
RUNNING → FAILED

WAITING → RUNNING
WAITING → CANCELLING
WAITING → FAILED

CANCELLING → CANCELLED
CANCELLING → FAILED
CANCELLING → SUCCEEDED
```

Переход `CANCELLING → SUCCEEDED` допускается, если операция успела достичь точки необратимого завершения до обработки запроса на отмену.

После перехода в `SUCCEEDED`, `FAILED` или `CANCELLED` состояние является терминальным и не изменяется.

Повторное выполнение должно создавать новую операцию.

## API длительных операций

Платформа должна предоставлять стандартный сервис управления операциями.

```proto
service Operations {
  rpc GetOperation(GetOperationRequest) returns (Operation);
  rpc ListOperations(ListOperationsRequest) returns (ListOperationsResponse);
  rpc CancelOperation(CancelOperationRequest) returns (google.protobuf.Empty);
  rpc DeleteOperation(DeleteOperationRequest) returns (google.protobuf.Empty);
  rpc WaitOperation(WaitOperationRequest) returns (Operation);
}
```

### GetOperation

Возвращает текущее состояние операции.

```proto
message GetOperationRequest {
  string name = 1;
}
```

Вызов должен быть безопасным и идемпотентным.

### ListOperations

Возвращает операции в рамках доступной пользователю области.

Поддерживаемые фильтры:

- состояние;
- тип операции;
- целевой ресурс;
- инициатор;
- время создания;
- время завершения;
- модуль;
- идентификатор корреляции.

Поддерживаемая сортировка:

- по времени создания;
- по времени обновления;
- по времени завершения.

Обязательна пагинация.

### CancelOperation

Регистрирует запрос на отмену.

Успешный ответ означает, что запрос на отмену принят, но не гарантирует немедленную отмену.

После вызова клиент должен продолжать проверять состояние операции.

Повторный вызов `CancelOperation` должен быть идемпотентным.

### DeleteOperation

Удаляет операцию из пользовательского представления или архива активных операций.

Удаление операции:

- не отменяет выполнение;
- не удаляет бизнес-ресурс;
- не удаляет обязательные аудиторские данные;
- не должно применяться к выполняющейся операции, если это запрещено политикой хранения.

В основной конфигурации M8 операции не удаляются физически через публичный API. `DeleteOperation` может быть реализован как скрытие операции из пользовательского списка либо не поддерживаться.

### WaitOperation

Ожидает изменения или завершения операции в пределах заданного срока.

```proto
message WaitOperationRequest {
  string name = 1;
  google.protobuf.Duration timeout = 2;
}
```

Сервер не обязан удерживать соединение до полного завершения. Он может вернуть текущее состояние после внутреннего ограничения ожидания.

## Запуск операции бизнес-командой

Метод, запускающий длительную операцию, возвращает `Operation`.

Пример:

```proto
rpc CreateProject(CreateProjectRequest) returns (google.longrunning.Operation) {
  option (google.longrunning.operation_info) = {
    response_type: "m8.resourcemanager.v1.Project"
    metadata_type: "m8.operations.v1.OperationMetadata"
  };
}
```

Начальный ответ:

```json
{
  "name": "operations/01JZ7A6K4WQH8Y3X2D5N9M1P7C",
  "metadata": {
    "@type": "type.googleapis.com/m8.operations.v1.OperationMetadata",
    "state": "PENDING",
    "stage": "VALIDATING",
    "message": "Запрос принят",
    "progressPercent": 0,
    "resourceName": "organizations/org-1/workspaces/ws-1/projects/project-1",
    "cancellable": true
  },
  "done": false
}
```

Успешное завершение:

```json
{
  "name": "operations/01JZ7A6K4WQH8Y3X2D5N9M1P7C",
  "metadata": {
    "@type": "type.googleapis.com/m8.operations.v1.OperationMetadata",
    "state": "SUCCEEDED",
    "stage": "COMPLETED",
    "message": "Проект создан",
    "progressPercent": 100
  },
  "done": true,
  "response": {
    "@type": "type.googleapis.com/m8.resourcemanager.v1.Project",
    "name": "organizations/org-1/workspaces/ws-1/projects/project-1",
    "displayName": "Production"
  }
}
```

## Идемпотентность

Команды, запускающие длительные операции, должны поддерживать идемпотентность.

Клиент передаёт ключ идемпотентности:

```text
Idempotency-Key: 018f7ad8-b89c-7a02-a7bd-4e7328fd12b6
```

Ключ должен быть уникален в рамках:

- инициатора;
- метода;
- области ресурса;
- установленного периода хранения.

При повторении идентичного запроса с тем же ключом сервис возвращает ранее созданную операцию.

Если с тем же ключом поступает запрос с другим содержимым, сервис возвращает ошибку `ALREADY_EXISTS` или `FAILED_PRECONDITION`.

Сервис должен хранить:

- ключ идемпотентности;
- хеш нормализованного запроса;
- имя операции;
- время создания;
- срок действия записи.

Идемпотентность запуска операции не означает идемпотентность каждого внутреннего шага. Все шаги процесса также должны проектироваться с учётом повторного выполнения.

## Создание операции и атомарность

Создание записи операции и фиксация намерения выполнить бизнес-команду должны происходить атомарно.

Рекомендуемая последовательность:

1. проверить авторизацию;
2. проверить синтаксическую и бизнес-валидацию;
3. проверить ключ идемпотентности;
4. создать запись операции;
5. зафиксировать начальное состояние целевого ресурса;
6. записать Outbox-событие запуска процесса;
7. завершить транзакцию;
8. вернуть операцию клиенту.

Недопустима ситуация, при которой клиент получил операцию, но информация о запуске процесса была потеряна.

Недопустима и обратная ситуация: процесс запущен, но операция не создана.

Для устранения разрыва между базой данных и брокером событий используется Transactional Outbox.

## Целевой ресурс

Если операция создаёт ресурс, сервис должен как можно раньше зарезервировать его идентификатор.

Пример:

```text
organizations/org-1/workspaces/ws-1/projects/project-1
```

Целевой ресурс может существовать в промежуточном состоянии:

```proto
enum ResourceState {
  RESOURCE_STATE_UNSPECIFIED = 0;
  CREATING = 1;
  ACTIVE = 2;
  UPDATING = 3;
  DELETING = 4;
  FAILED = 5;
}
```

Это позволяет:

- ссылаться на ресурс до завершения операции;
- исключать создание дубликатов;
- блокировать конфликтующие команды;
- отображать ресурс в интерфейсе;
- сохранять информацию об ошибке;
- повторно запускать восстановление.

Операция и ресурс имеют разные жизненные циклы.

Например, операция создания может перейти в `FAILED`, а ресурс остаться в состоянии `FAILED` для диагностики и повторного запуска.

## Оркестрация

### Простые операции

Простая операция может выполняться фоновым обработчиком сервиса, если она:

- имеет один или два локальных этапа;
- не требует длительного ожидания;
- не требует сложной компенсации;
- может быть полностью восстановлена на основе записи в базе данных.

### Сложные операции

Temporal должен использоваться, если операция:

- включает несколько сервисов;
- длится минуты, часы или дни;
- ожидает внешние события;
- требует таймеров;
- требует повторных попыток;
- содержит компенсирующие действия;
- требует сигналов отмены;
- должна гарантированно продолжаться после перезапуска;
- включает ручное подтверждение.

Одна пользовательская операция обычно соответствует одному экземпляру Temporal Workflow.

Пример идентификатора:

```text
operation:01JZ7A6K4WQH8Y3X2D5N9M1P7C
```

Workflow ID должен быть детерминированно связан с именем операции.

## Разделение Operation и Workflow

Публичная операция и Temporal Workflow не являются одним объектом.

`Operation`:

- является частью публичного API;
- хранит пользовательское состояние;
- содержит прогресс и результат;
- применяется для авторизации доступа;
- сохраняется в хранилище платформы;
- имеет установленную политику хранения.

`Workflow`:

- является внутренним механизмом исполнения;
- содержит техническую историю;
- не предоставляется непосредственно клиенту;
- может продолжаться через Continue-As-New;
- может иметь собственную политику хранения.

Клиент не должен зависеть от идентификаторов, состояний и терминологии Temporal.

## Шаги выполнения

Каждая сложная операция должна быть разбита на явно определённые шаги.

Пример создания управляемого сервиса:

```text
1. ValidateRequest
2. ReserveResource
3. SelectPlacement
4. AllocateQuota
5. CreateProviderResource
6. WaitProviderReady
7. ConfigureNetwork
8. ConfigureCredentials
9. VerifyHealth
10. ActivateResource
11. PublishCompletionEvent
```

Для каждого шага должны быть определены:

- входные данные;
- выходные данные;
- максимальное время выполнения;
- политика повторных попыток;
- признак идемпотентности;
- возможные ошибки;
- возможность отмены;
- компенсирующее действие;
- обновление прогресса;
- метрики.

## Повторные попытки

Повторные попытки применяются только для временных ошибок.

Примеры временных ошибок:

- недоступность сети;
- превышение времени ожидания;
- временная недоступность внешнего API;
- ограничение частоты запросов;
- временная блокировка;
- ошибка лидера или раздела хранилища;
- недоступность брокера.

Примеры постоянных ошибок:

- неверная конфигурация;
- отсутствие прав;
- несуществующий обязательный ресурс;
- нарушение бизнес-ограничения;
- неподдерживаемый тип ресурса;
- исчерпание невосстанавливаемой квоты;
- отклонение запроса внешним провайдером.

Базовая политика:

```text
initial_interval: 1s
backoff_coefficient: 2.0
maximum_interval: 1m
maximum_attempts: 10
```

Конкретная политика определяется для каждого шага.

Повторные попытки не должны применяться без ограничений.

## Таймауты

Необходимо различать:

- таймаут клиентского запроса;
- таймаут ожидания операции;
- таймаут отдельного шага;
- общий срок выполнения операции;
- таймаут взаимодействия с внешней системой;
- срок ожидания ручного действия.

Завершение клиентского запроса по таймауту не отменяет операцию.

Для каждой операции должен быть установлен максимальный срок выполнения либо политика бессрочного ожидания, явно разрешённая архитектурой.

Истечение общего срока может приводить к:

- завершению с ошибкой;
- переходу к компенсации;
- переходу в состояние ожидания вмешательства;
- созданию инцидента;
- постановке операции в очередь ручного разбора.

## Отмена

Отмена является кооперативной.

Запрос на отмену устанавливает признак:

```text
cancel_requested = true
```

Исполнитель проверяет запрос:

- между шагами;
- перед необратимыми действиями;
- во время длительного ожидания;
- при получении сигнала Temporal.

Для каждой операции должна быть определена точка необратимости.

До точки необратимости операция может быть отменена с компенсацией.

После точки необратимости:

- отмена может быть отклонена;
- операция может завершиться успешно;
- может быть запущена отдельная операция удаления или отката.

Поле `cancellable` должно динамически отражать возможность отмены.

Ответ на `CancelOperation` не должен содержать ложную гарантию отмены.

## Компенсация

Распределённая операция не должна рассчитывать на глобальную транзакцию.

Для отмены частично выполненных изменений используется Saga.

Пример:

```text
CreateProviderResource
    compensation: DeleteProviderResource

AllocateQuota
    compensation: ReleaseQuota

CreateCredentials
    compensation: RevokeCredentials
```

Компенсация должна быть:

- идемпотентной;
- наблюдаемой;
- повторяемой;
- ограниченной по количеству попыток;
- защищённой от удаления чужого ресурса;
- связанной с исходной операцией.

Если компенсация не завершилась, операция переходит в `FAILED`, а метаданные должны содержать предупреждение о необходимости ручного вмешательства.

Нельзя сообщать клиенту, что операция полностью отменена, если часть изменений осталась активной.

## Обработка ошибок

Итоговая ошибка представляется через `google.rpc.Status`.

```proto
message Status {
  int32 code = 1;
  string message = 2;
  repeated google.protobuf.Any details = 3;
}
```

Ошибка должна содержать:

- стабильный код;
- безопасное пользовательское сообщение;
- технический идентификатор ошибки;
- сведения о нарушениях полей;
- сведения о повторной попытке, если она допустима;
- ссылку на проблемный ресурс;
- сведения о частично выполненных действиях.

В `message` запрещено передавать:

- стек вызовов;
- токены;
- пароли;
- внутренние адреса;
- секреты;
- персональные данные без необходимости;
- полные ответы внешних провайдеров.

Пример:

```json
{
  "code": 8,
  "message": "Недостаточно квоты для создания ресурса",
  "details": [
    {
      "@type": "type.googleapis.com/google.rpc.ResourceInfo",
      "resourceType": "m8.platform.v1.Quota",
      "resourceName": "organizations/org-1/quotas/managed-kafka"
    }
  ]
}
```

## Частичный результат

По умолчанию операция либо завершается полностью, либо возвращает ошибку.

Частичный результат допускается для:

- массовых операций;
- импорта;
- экспорта;
- пакетной обработки;
- миграций;
- операций над независимыми ресурсами.

В таком случае результат должен явно содержать:

- количество успешных элементов;
- количество неуспешных элементов;
- ошибки по каждому элементу;
- идентификатор отчёта;
- возможность повторить только неуспешные элементы.

Нельзя использовать состояние `SUCCEEDED`, если основной ожидаемый эффект операции не был достигнут, кроме явно описанных пакетных операций с допустимым частичным успехом.

## Конкурентное выполнение

Сервис должен предотвращать конфликтующие операции над одним ресурсом.

Примеры конфликтов:

- обновление ресурса во время удаления;
- два параллельных изменения одной конфигурации;
- повторное создание уже создаваемого ресурса;
- перенос ресурса во время изменения его параметров.

Возможные механизмы:

- оптимистическая блокировка по версии;
- lease на ресурс;
- запись активной операции в ресурсе;
- сериализация Workflow по идентификатору ресурса;
- отдельный диспетчер команд;
- проверка предварительных условий.

Рекомендуемое поле ресурса:

```proto
string active_operation = 20;
```

При конфликте сервис возвращает `FAILED_PRECONDITION` или существующую активную операцию.

## Версионирование ресурса

Команды изменения должны поддерживать контроль версии.

Пример:

```proto
message UpdateProjectRequest {
  Project project = 1;
  google.protobuf.FieldMask update_mask = 2;
  uint64 expected_version = 3;
  string request_id = 4;
}
```

Версия проверяется до создания операции.

Для операций с длительным ожиданием версия может быть повторно проверена перед критическим изменением.

Если ресурс был изменён другой командой, операция должна:

- завершиться с конфликтом;
- перестроить план, если это безопасно;
- запросить ручное разрешение конфликта.

Автоматическое перезаписывание изменений запрещено.

Для выборки операций необходимы индексы:

- по имени операции;

## Согласованность

Состояние операции является пользовательской проекцией фактического выполнения.

Обновление операции может быть согласовано с Workflow с небольшой задержкой.

Требования:

- терминальное состояние не должно откатываться;
- прогресс не должен уменьшаться;
- события обновления должны быть упорядочены по версии;
- повторно доставленное событие не должно повреждать состояние;
- устаревшее обновление должно игнорироваться;
- результат должен записываться только после фиксации бизнес-изменений.

Каждое обновление операции должно содержать монотонно растущую версию.

```proto
message OperationEvent {
```

События предназначены для:

- интерфейса пользователя;

- аудита;

- построения проекций;
- контроля SLO.

Событие завершения должно публиковаться через Outbox после фиксации итогового состояния.

## Авторизация

Доступ к операции определяется доступом к целевому ресурсу и политикой модуля.

Проверки должны выполняться при каждом вызове:

- `GetOperation`;
- `ListOperations`;
- `CancelOperation`;
- `WaitOperation`.

Создатель операции не получает автоматического бессрочного доступа, если доступ к организации или ресурсу был отозван.

Для операций, ещё не связанных с созданным ресурсом, используется область родительского ресурса.

Пример:

```text
organizations/org-1/workspaces/ws-1
```

Для отмены может требоваться отдельное разрешение:

```text
operations.cancel
```

Просмотр технических деталей ошибок может требовать:

```text
operations.debug
```

## Аудит

В аудит записываются:

- создание операции;
- инициатор;
- исходная команда;
- целевой ресурс;
- ключ идемпотентности;
- переходы состояния;
- изменение этапа;
- запрос на отмену;
- результат отмены;
- итоговый результат;
- итоговая ошибка;
- ручное вмешательство;
- повторный запуск;
- компенсирующие действия;
- административное скрытие или архивирование.

Аудит не должен содержать секретные значения из запроса или ответа.

Записи аудита являются неизменяемыми.

## Наблюдаемость

Для каждой операции должны быть доступны:

- `operation_id`;
- `workflow_id`;
- `trace_id`;
- `correlation_id`;
- `resource_name`;
- `operation_type`;
- `requested_by`;
- текущий этап;
- количество попыток;
- длительность;
- итоговый статус.

### Метрики

Обязательные метрики:

```text

```

Основные разрезы:

- модуль;
- тип операции;
- состояние;
- этап;
- код ошибки;
- внешний провайдер;
- регион;
- класс ресурса.

Нельзя добавлять `operation_id` в labels метрик из-за высокой кардинальности.

### Логи

Каждая запись лога должна содержать:

```text

```

### Трассировка

Операция должна иметь корневой span либо ссылку на исходный trace.

Переход между API, Outbox, брокером, Workflow и Activity должен сохранять контекст трассировки.

## Выявление зависших операций

Операция считается потенциально зависшей, если:

- `update_time` не изменялось дольше допустимого интервала;
- шаг превысил установленный таймаут;
- Workflow отсутствует, но операция активна;
- Workflow завершён, но операция не переведена в терминальное состояние;
- количество повторных попыток превышает порог;
- ожидание внешней системы превысило установленный срок;
- компенсация не завершилась.

Для таких операций должна выполняться периодическая сверка.

Возможные действия:

- восстановить Workflow;
- повторно запустить безопасный шаг;
- синхронизировать состояние;
- завершить операцию с ошибкой;
- отправить уведомление;
- создать инцидент;
- перевести операцию в ручной разбор.

## Восстановление после сбоя

После перезапуска сервиса:

- все активные операции должны оставаться доступными;
- Workflow должен продолжить выполнение;
- необработанные Outbox-события должны быть опубликованы;
- повторная доставка событий не должна создавать дубликаты;
- незавершённые локальные задания должны быть переподхвачены;
- блокировки с истёкшим lease должны быть освобождены или восстановлены.

In-memory очередь не может быть единственным хранилищем состояния операции.

## Политика хранения

Операции хранятся дольше, чем техническая история исполнения.

Рекомендуемая политика:

- активные операции — без ограничения до завершения;
- завершённые операции — не менее 90 дней;
- операции, связанные с безопасностью и доступом, — согласно политике аудита;
- технические payload — могут быть очищены раньше;
- аудит — хранится отдельно по установленным требованиям;
- персональные данные — минимизируются и удаляются согласно политике хранения.

После истечения срока операция может быть:

- перемещена в архив;
- обезличена;
- исключена из обычного списка;
- удалена физически, если это допускается политикой.

## Пользовательский интерфейс

Интерфейс должен отображать:

- название операции;
- целевой ресурс;
- состояние;
- этап;
- прогресс;
- время начала;
- длительность;
- инициатора;
- предупреждения;
- итоговый результат;
- безопасное описание ошибки;
- возможность отмены;
- ссылку на созданный ресурс;
- ссылку на журнал аудита при наличии прав.

Интерфейс не должен предполагать, что все операции имеют точный процент выполнения.

Для активных операций должна поддерживаться автоматическая актуализация:

- периодическим запросом;
- Server-Sent Events;
- WebSocket;
- подпиской на внутреннюю проекцию событий.

Периодический запрос должен использовать увеличение интервала:

```text
1s → 2s → 5s → 10s → 30s
```

## SLO

Для механизма длительных операций устанавливаются следующие базовые цели:

- 99,9% запросов создания операции подтверждаются не более чем за 1 секунду;
- 99,9% запросов `GetOperation` выполняются не более чем за 500 мс;
- 99,99% принятых операций не теряются;
- не менее 99,9% обновлений терминального состояния становятся доступны клиенту в течение 10 секунд после завершения процесса;
- 100% терминальных операций содержат либо результат, либо структурированную ошибку;
- 100% операций имеют идентификатор корреляции;
- 100% запросов на отмену фиксируются в аудите.

SLO конкретных бизнес-операций задаются отдельно.

## Norms

1. Все длительные команды возвращают объект `google.longrunning.Operation`.
2. Каждая операция имеет глобально уникальное и неизменяемое имя.
3. Запуск операции должен поддерживать ключ идемпотентности.
4. Создание операции и фиксация намерения выполнения должны быть атомарными.
5. Активная операция должна переживать перезапуск сервиса.
6. Терминальное состояние операции не может быть изменено.
7. Успешная операция обязана содержать типизированный результат.
8. Неуспешная операция обязана содержать `google.rpc.Status`.
9. Прогресс не должен уменьшаться.
10. Пользовательское сообщение не используется как машинный код состояния.
11. Отмена является кооперативной и не гарантирует откат после точки необратимости.
12. Каждый внутренний шаг должен быть идемпотентным или защищённым от повторного выполнения.
13. Временные и постоянные ошибки должны обрабатываться по-разному.
14. Повторные попытки должны иметь ограничения и экспоненциальную задержку.
15. Для сложных многоэтапных процессов используется Temporal.
16. Для согласования базы данных и брокера используется Transactional Outbox.
17. Глобальные распределённые транзакции между сервисами не применяются.
18. Для отката частично выполненных действий используются компенсации.
19. Все переходы состояния фиксируются в аудите.
20. Все операции должны быть наблюдаемыми через метрики, логи и трассировку.
21. Секреты и внутренние технические детали не возвращаются клиенту.
22. Операция и целевой бизнес-ресурс имеют независимые жизненные циклы.
23. Удаление записи операции не должно отменять бизнес-процесс.
24. Конфликтующие операции над одним ресурсом должны сериализоваться или отклоняться.
25. Состояние операции должно обновляться с использованием оптимистической блокировки или версии.

## Safeguards

### Защита от повторного запуска

- обязательный `Idempotency-Key`;
- хранение хеша запроса;
- детерминированный Workflow ID;
- уникальное ограничение на активную операцию соответствующего типа;
- дедупликация входящих событий.

### Защита от двойного применения шага

- идемпотентные API провайдеров;
- передача внешнего request ID;
- проверка существования ресурса перед созданием;
- compare-and-set;
- Inbox для событий;
- таблица выполненных шагов.

### Защита от зависания

- таймауты шагов;
- heartbeat;
- watchdog;
- периодическая сверка Operation и Workflow;
- максимальный срок ожидания;
- уведомления и автоматическое создание инцидента.

### Защита от неконтролируемых повторных попыток

- максимальное количество попыток;
- экспоненциальная задержка;
- jitter;
- Circuit Breaker;
- ограничение параллелизма;
- Dead Letter Queue для необрабатываемых сообщений.

### Защита от перегрузки

- квоты на количество активных операций;
- ограничение по организации;
- ограничение по типу ресурса;
- приоритеты;
- очереди;
- backpressure;
- rate limiting;
- отдельные worker pools для разных классов операций.

### Защита данных

- шифрование чувствительных payload;
- исключение секретов из `metadata`;
- очистка технических ответов провайдеров;
- ограниченный срок хранения payload;
- авторизация на каждый запрос;
- аудит доступа к результатам.

### Защита от конфликтов

- версия ресурса;
- precondition;
- активная операция на ресурсе;
- lease;
- сериализация команд;
- повторная проверка состояния перед необратимым шагом.

### Защита от ошибочной отмены

- проверка права `operations.cancel`;
- фиксация инициатора отмены;
- динамический признак `cancellable`;
- точка необратимости;
- подтверждение для критических операций;
- отдельная операция отката после необратимого изменения.

## Антипаттерны

Запрещаются следующие подходы:

### Фоновая goroutine без устойчивого состояния

```go
go createResource(ctx, request)
```

После перезапуска процесс будет потерян.

### Использование HTTP-соединения как состояния процесса

Разрыв соединения не должен определять судьбу бизнес-операции.

### Неограниченные повторные попытки

Они могут привести к бесконечной нагрузке и повторному применению изменений.

### Сохранение состояния только в Temporal

Публичная операция должна иметь независимую пользовательскую проекцию и собственную модель авторизации.

### Возврат только идентификатора задания

Клиент должен получать стандартный объект операции с состоянием, метаданными, ошибкой и результатом.

### Использование произвольных строковых статусов

Состояния и этапы должны иметь стабильные перечисления или коды.

### Удаление частично созданного ресурса без аудита

Компенсация является частью операции и должна быть наблюдаемой.

### Объявление отмены до завершения компенсации

Состояние `CANCELLED` допускается только после достижения согласованного состояния.

### Повторное открытие завершённой операции

Для нового выполнения создаётся новая операция.

## Пример: создание проекта

### Запрос

```proto
message CreateProjectRequest {
  string parent = 1;
  string project_id = 2;
  Project project = 3;
  string request_id = 4;
}
```

### Сценарий

```text
1. Проверить право projects.create.
2. Проверить request_id.
3. Проверить уникальность project_id.
4. Создать Project в состоянии CREATING.
5. Создать Operation в состоянии PENDING.
6. Записать Outbox-событие ProjectCreationRequested.
7. Запустить Workflow.
8. Настроить пространство имён и политики доступа.
9. Создать базовые системные роли.
10. Перевести Project в ACTIVE.
11. Завершить Operation с Project в response.
12. Опубликовать ProjectCreated и OperationSucceeded.
```

### Ошибка

Если создание базовых политик доступа завершилось ошибкой:

- Workflow выполняет повторные попытки;
- при исчерпании попыток запускается компенсация;
- проект переводится в `FAILED` либо удаляется согласно политике;
- операция переходит в `FAILED`;
- ошибка содержит идентификатор проекта и безопасную причину.

## Пример: удаление организации

Удаление организации является длительной и потенциально необратимой операцией.

Этапы:

```text
VALIDATING
BLOCKING_NEW_OPERATIONS
DISCOVERING_RESOURCES
DELETING_DEPENDENT_RESOURCES
WAITING_RESOURCE_DELETION
DELETING_IDENTITY_DATA
DELETING_ACCESS_RELATIONS
ANONYMIZING_AUDIT_REFERENCES
FINALIZING
COMPLETED
```

Особые требования:

- обязательное подтверждение;
- проверка отсутствия юридической блокировки;
- формирование плана удаления;
- отдельная точка необратимости;
- прекращение возможности отмены после начала физического удаления данных;
- сохранение обязательного аудита;
- формирование итогового отчёта;
- защита от повторного удаления;
- ограничение параллельного выполнения.

## Решения по умолчанию для M8 Platform

Для платформы M8 принимаются следующие решения:

| Область | Решение |
| --- | --- |
| Публичная модель | `google.longrunning.Operation` |
| Оркестрация | Temporal |
| Основной язык исполнителей | Go |
| Хранилище операций | YDB |
| Публикация событий | Kafka или YDB Topics |
| Согласование БД и событий | Transactional Outbox |
| Дедупликация событий | Inbox |
| Контроль конкурентности | Version VO и активная операция ресурса |
| Ошибки | `google.rpc.Status` |
| Валидация | Protobuf Validation |
| Наблюдаемость | OpenTelemetry, Prometheus, Grafana |
| Аудит | M8 Audit |
| Авторизация | M8 Access |
| Идентификатор операции | ULID |
| Идентификатор Workflow | На основе имени операции |
| Удаление операции | Физическое удаление через API не поддерживается |
| Отмена | Temporal Signal и кооперативная компенсация |

## Критерии готовности

Механизм длительных операций считается готовым, если:

- определены Protobuf-контракты;
- реализованы `Get`, `List`, `Cancel` и `Wait`;
- реализована идемпотентность запуска;
- операция создаётся атомарно с Outbox-записью;
- реализована связь с Temporal Workflow;
- определены состояния и переходы;
- реализовано обновление прогресса;
- реализовано структурированное завершение;
- реализована кооперативная отмена;
- реализована минимум одна компенсация;
- настроены аудит и трассировка;
- настроены метрики и предупреждения о зависших операциях;
- реализована сверка Operation и Workflow;
- проведены тесты повторной доставки;
- проведены тесты перезапуска;
- проведены тесты отмены на каждом этапе;
- проведены тесты отказа внешнего провайдера;
- проведены тесты параллельных конфликтующих команд;
- документирована точка необратимости для каждой критической операции.
