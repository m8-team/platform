# Domain Model

## PlatformInstallation

Fields: `metadata`, `spec.platformVersion`, `spec.profile`, infrastructure sections, module flags, upgrade, air-gap, status.

Invariants:

- `spec.platformVersion` is required.
- `production` requires HA assumptions, backup, external secrets and signature enforcement.
- `air-gapped` requires private registry coordinates.
- CNI replacement requires explicit `allowCNIReplacement=true`.
- Enabled modules must have their infrastructure dependencies.

Lifecycle: `Pending -> Validating -> Planning -> Bootstrapping -> Installing -> Verifying -> Ready`; degraded and failed states are terminal for the current operation but resumable through a new `InstallationOperation`.

## PlatformRelease

Fields: Kubernetes compatibility, component release map, artifact references, migration requirements, rollback metadata, security metadata, signature.

Invariants:

- No `latest`, `main` or `master`.
- Charts and images must be digest-pinned.
- Production use requires a release signature.
- Upgrade paths and non-rollbackable boundaries must be explicit.

Lifecycle: authored by release engineering, signed, imported into connected or air-gapped catalog, then referenced by `PlatformInstallation`.

## ComponentRelease

Fields: version, chart, images, CRD versions, dependencies, upgrade paths, rollback support, migrations, resource estimates, SBOM and signature metadata.

Invariant: every executable artifact is immutable by digest.

Lifecycle: created as part of a `PlatformRelease`; consumed by planner and bundle exporter/importer.

## InstallationPlan

Fields: installation ref, release ref, config digest, release digest, profile, ordered steps, risks, irreversible actions, resource estimate.

Invariants:

- Plan order is produced from a DAG.
- Plan includes digests for both source config and release catalog.
- A saved plan is applyable only if those digests still match.

Lifecycle: generated by `m8ctl plan`; later signed and executed by `m8ctl install --plan`.

## InstallationOperation

Fields: operation ID, type, installation ref, requested version, plan digest, initiator, phase, current step, completed steps, retry count, checkpoints, error, logs ref.

Invariants:

- Operation ID is unique.
- Irreversible checkpoints cannot be replayed as normal rollback steps.
- Transient and permanent errors are distinguished.

Lifecycle: created at bootstrap/install/upgrade/backup/restore/rollback/uninstall start; updated after each verified checkpoint.

## InstallationStep

Fields: ID, wave, title, phase, dependencies, change set, readiness gates, rollback policy.

Invariants:

- ID is unique within a plan.
- Dependencies must exist.
- Readiness must be observed, not time-slept.

Lifecycle: planned, applied, verified, checkpointed, and optionally rolled back.

## CheckResult

Fields: ID, category, severity, status, message, details, remediation, documentation reference, duration.

Invariants:

- Every failed check has remediation.
- Details must not contain secrets.
- Machine-readable output preserves stable IDs.

Lifecycle: emitted by `preflight` and `doctor`; aggregated into reports.

## ChangeSet

Fields: namespaces, CRDs, Helm releases, Argo CD Applications, external checks, secrets, certificates, routes, policies, migrations, smoke tests.

Invariant: a change set describes intended changes and does not execute them.

Lifecycle: produced by planner; consumed by install/upgrade executors.

## Backup

Fields: installation ref, component list, storage ref, verification flag, artifacts, platform version, status.

Invariants:

- Velero is not considered a database-native backup.
- Component artifacts must carry digest or provider-native immutable version IDs.

Lifecycle: requested, component backups created, metadata captured, verified.

## RestorePlan

Fields: installation ref, backup ref, source and target versions, replacement resources, maintenance actions, risks, expected RPO/RTO.

Invariants:

- Restore compatibility must be checked before execution.
- Functional smoke tests are mandatory after restore.

Lifecycle: planned, reviewed, approved, executed, verified.

## DiagnosticBundle

Fields: statuses, events, sanitized manifests, CRDs, versions, logs, health checks, network diagnostics, resource usage, installation metadata.

Invariants:

- Secrets, tokens, passwords and private keys are redacted before archive write.
- Bundle is immutable after creation and includes a checksum.

Lifecycle: created by `m8ctl doctor --bundle`, attached to support cases or incident records.

## AirGapBundle

Fields: images, Helm charts, CRDs, manifests, release catalog, SBOMs, signatures, policies, dashboards, alerts, migrations, docs, checksums, license data.

Invariants:

- Bundle completeness is verified before import.
- Imported artifacts must preserve original digest mapping.

Lifecycle: exported in connected environment, signed, transferred, verified, imported into private registries.
